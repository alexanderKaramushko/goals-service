## Техническое задание (ТЗ) для проекта "Социальная система целей и наград"

## 1. Общее описание

Создать веб-приложение, в котором пользователи могут:
- создавать цели
- прикреплять промежуточные шаги к целям
- просматривать цели других пользователей
- прикреплять награды к завершенным целям
- прикреплять награды к пользователям, у которых достаточно звёзд

Другие пользователи также могут идеть статистику по баллам и штрафам.

## 2. Основные функции

**2.1 Регистрация и вход**

Пользователь должен иметь возможность зарегистрироваться и войти в систему. <br />
Аутентификация реализуется через JWT либо через сторонние OAuth-провайдеры. <br />

**2.2 Управление целями**

Пользователь может создавать **цель** <br />
Пользователь может создавать промежуточныe шаги у **цели** <br />

Пользователь может редактировать или отменить цель. <br />
Отменённая цель считается закрытой с вычетом звёзд. <br />

**2.3 Обработка целей**

За выполнение цели пользователь получает звёзды. <br />
За отмену цели — штраф: списывает звёзды у пользователя. <br />

**2.4 Просмотр целей**

Пользователь может просматривать список целей других пользователей. <br />

В списке отображаются:

- Статистика по количеству достигнутых целей и счет звёзд
- Возможность создать **награду** для завершенной цели, если не превышен лимит по просроченным шагам
- Возможность создать большую награду для другого пользователя, если у него есть нужное кол-во баллов

**2.5 Создание награды**

Пользователь может создать награду:

- Для закрытой цели другого пользователя, если она будет закрыта без 20% (далее будет детализировано) штрафов
- За определённое количество полученных звёзд за закрытые цели

**2.6 Статистика и награды**

За выполнение каждой цели пользователь получает звёзды (например, +10 звёзд). <br />

В профиле отображается:
- Общее число достигнутых целей
- Общее число полученных звёзд

## 3. Модели данных (актуально по SQL-миграциям)

**3.0 Концептуальная модель**

**Сущности**: `users`, `targets`, `steps`, `rewards`.

**ER-диаграмма**:

```mermaid
erDiagram
    USERS {
      string id PK
      string full_name
      timestamp created_at "TIMESTAMPTZ, DEFAULT now(), NULL"
    }
    TARGETS {
      int id PK
      string user_id FK
      string title
      string description
      string status "NULL, CHECK (status IN ('created', 'active', 'completed', 'cancelled'))"
      date should_be_completed_at
      date completed_at "NULL"
      timestamp closed_at "NULL"
      timestamp created_at "NULL"
      timestamp updated_at "NULL"
    }
    STEPS {
      int id PK
      int target_id FK "NULL, ON DELETE SET NULL"
      string title
      string description
      date should_be_completed_at
      timestamp closed_at "NULL"
      timestamp created_at "DEFAULT now()"
      date completed_at "NULL"
    }
    REWARDS {
      int id PK
      string user_id FK "NULL"
      int target_id FK "NULL"
      string title
      string description
      string type "CHECK (type IN ('target', 'user'))"
      timestamp created_at
      timestamp accepted_at "NULL"
    }

    USERS ||--o{ TARGETS : has
    TARGETS |o--o{ STEPS : has
    TARGETS |o--o{ REWARDS : reward_for_target
    USERS |o--o{ REWARDS : reward_for_user
```

> Примечание: в текущих миграциях таблица наград называется `surprises`.

**3.4 Статистика**
Собирается динамически из данных целей, звёзд, наград. <br />

### 4. Логика бизнес-процессов

**Создание цели**: пользователь заходит в систему, создает цель, время, когда нужно ее завершить добавляет шаги (или без шагов) и нажимает "Старт". <br />
**Назначение награды к цели**: другой пользователь заходит в систему, добавляет награду к завершенной цели,если она была закрыта без превышения лимитов по штрафам. <br />
**Назначение награды за баллы**: другой пользователь заходит в систему, переходит в профиль пользователя <br />
и создает награду за звёзды, у другого пользователя списываются звёзды, если он принимает награду. <br />
**Отмена цели**: пользователь нажимает кнопку "Отменить", фиксируется дата отмены, списываются звёзды. <br />
**Выполнение шага**: когда наступает дата выполнения шага, то пользователь должен записать итог шага и пометить его как выполненный. <br />
**Получение наград**: после достижения цели пользователю начисляются звёзды. <br />

### 5. Технические требования

Backend: NestJS с использованием PostgreSQL (`pg` + SQL-миграции через `node-pg-migrate`). <br />
Аутентификация: JWT либо сторонние OAuth-провайдеры. <br />
API: RESTful. <br />
Безопасность: защита эндпоинтов, проверка прав доступа. <br />
Миграции/обновление схем: при необходимости. <br />

### 6. Дополнительные требования

Возможность просмотра списка целей в ЛК. <br />
Возможность баллов в ЛК. <br />
Фильтрация целей по статусу, дате, автору. <br />
